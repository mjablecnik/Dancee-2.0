# Requirements Document

## Introduction

This feature adds Firebase Authentication to the Dancee App ecosystem. Currently, the app uses a hardcoded `defaultUserId` for favorites and has no real authentication. The auth screens (login, register, forgot password, onboarding) exist as UI shells with no backend logic.

This feature introduces:
- Firebase Authentication with email/password, Google Sign-In, and Apple Sign-In
- Email verification after registration
- Password reset via Firebase
- Auth state management (AuthCubit) integrated with the existing Cubit/Bloc architecture
- Protected routing based on auth state (unauthenticated → login, authenticated → events)
- Firebase-Directus user synchronization so that CMS operations (favorites, etc.) use the real Firebase UID instead of a hardcoded default
- An email verification screen (new) and wiring of existing auth screens to Firebase
- All new UI strings translated in en, cs, es via slang

### Existing Screens (UI shells, need Firebase logic wiring)
- LoginScreen (`/login`) — email/password fields, Google/Apple buttons, forgot password link
- RegisterScreen (`/register`) — name, email, password fields, Google/Apple buttons, terms checkbox
- ForgotPasswordScreen (`/forgot-password`) — email field, send reset link button
- OnboardingScreen (`/onboarding`) — 3-step wizard (dance styles, level, location)

### New Screens Needed
- EmailVerificationScreen (`/verify-email`) — prompts user to verify email, resend link, check status

## Glossary

- **App**: dancee_app2 — the Flutter frontend application
- **Firebase_Auth**: Firebase Authentication SDK — the client-side library handling sign-in, sign-up, token management, and identity provider integration
- **Auth_Repository**: A Dart class in the App that wraps Firebase_Auth SDK calls and exposes auth operations to cubits
- **Auth_Cubit**: A Cubit managing authentication state (unauthenticated, loading, authenticated, error) and coordinating auth operations
- **CMS**: dancee_cms — Directus headless CMS providing the REST API for data storage
- **Firebase_User**: The user object returned by Firebase_Auth after successful authentication, containing uid, email, displayName, emailVerified, and photoURL
- **Directus_User**: A record in the CMS `directus_users` collection representing the authenticated user, linked by Firebase UID
- **Auth_State**: An immutable state object representing the current authentication status (unauthenticated, loading, authenticated with user data, error with message)
- **Router_Guard**: A GoRouter redirect function that checks Auth_State and redirects unauthenticated users to login and authenticated users away from auth screens
- **Email_Verification_Screen**: A new screen shown after registration prompting the user to verify their email address
- **Onboarding_Screen**: An existing multi-step wizard screen shown after first-time registration to collect user preferences (dance styles, level, location)

## Requirements

### Requirement 1: Firebase SDK Configuration

**User Story:** As a developer, I want Firebase configured in the Flutter app and platform projects, so that Firebase Authentication services are available at runtime.

#### Acceptance Criteria

1. THE App SHALL include the `firebase_core`, `firebase_auth`, `google_sign_in`, and `sign_in_with_apple` packages as dependencies
2. THE App SHALL initialize Firebase in the `main()` function before `runApp()` using `Firebase.initializeApp()` with platform-specific configuration generated by FlutterFire CLI
3. THE App SHALL store Firebase configuration files (`google-services.json` for Android, `GoogleService-Info.plist` for iOS) in the appropriate platform directories
4. THE App SHALL add the Firebase project ID and web API key to `lib/config.dart` (gitignored) and provide placeholder values in `lib/config.example.dart`
5. IF Firebase initialization fails, THEN THE App SHALL display an error screen with a descriptive message instead of crashing

### Requirement 2: Auth Repository

**User Story:** As a developer, I want a centralized auth repository wrapping Firebase Auth SDK calls, so that all authentication logic is encapsulated in one place.

#### Acceptance Criteria

1. THE Auth_Repository SHALL provide a method to sign in with email and password using Firebase_Auth
2. THE Auth_Repository SHALL provide a method to register a new user with email, password, first name, and last name using Firebase_Auth, and set the displayName on the Firebase_User profile
3. THE Auth_Repository SHALL provide a method to sign in with Google using the `google_sign_in` package and Firebase_Auth credential
4. THE Auth_Repository SHALL provide a method to sign in with Apple using the `sign_in_with_apple` package and Firebase_Auth credential
5. THE Auth_Repository SHALL provide a method to send a password reset email via Firebase_Auth
6. THE Auth_Repository SHALL provide a method to send an email verification link to the current Firebase_User
7. THE Auth_Repository SHALL provide a method to reload the current Firebase_User and return the updated emailVerified status
8. THE Auth_Repository SHALL provide a method to sign out the current user from Firebase_Auth (and from Google Sign-In if applicable)
9. THE Auth_Repository SHALL expose a `Stream<Firebase_User?>` from Firebase_Auth `authStateChanges()` for reactive auth state observation
10. IF any Firebase_Auth operation throws a `FirebaseAuthException`, THEN THE Auth_Repository SHALL map the error code to a user-friendly error message string

### Requirement 3: Auth State Management

**User Story:** As a developer, I want an AuthCubit managing authentication state, so that the UI reacts to auth changes consistently across the app.

#### Acceptance Criteria

1. THE Auth_Cubit SHALL maintain an Auth_State with four variants: unauthenticated, loading, authenticated (containing Firebase_User data: uid, email, displayName, emailVerified), and error (containing an error message)
2. THE Auth_Cubit SHALL listen to the Auth_Repository `authStateChanges` stream and emit corresponding Auth_State updates
3. WHEN the Auth_Cubit receives a non-null Firebase_User from the stream, THE Auth_Cubit SHALL emit an authenticated state with the user data
4. WHEN the Auth_Cubit receives a null user from the stream, THE Auth_Cubit SHALL emit an unauthenticated state
5. THE Auth_Cubit SHALL provide methods for: signInWithEmail, signInWithGoogle, signInWithApple, register, sendPasswordReset, sendEmailVerification, reloadUser, and signOut — each delegating to the Auth_Repository
6. WHILE an auth operation is in progress, THE Auth_Cubit SHALL emit a loading state
7. IF an auth operation fails, THEN THE Auth_Cubit SHALL emit an error state with the user-friendly error message from the Auth_Repository

### Requirement 4: Email/Password Login

**User Story:** As a dancer, I want to sign in with my email and password, so that I can access my personalized dance event experience.

#### Acceptance Criteria

1. WHEN the user submits the login form with a valid email and password, THE App SHALL call Auth_Cubit signInWithEmail and authenticate via Firebase_Auth
2. WHEN login succeeds and the user's email is verified, THE App SHALL navigate to the events page (`/events`)
3. WHEN login succeeds and the user's email is not verified, THE App SHALL navigate to the email verification screen (`/verify-email`)
4. IF login fails due to invalid credentials, THEN THE App SHALL display an error message on the login screen without navigating away
5. IF login fails due to a disabled account, THEN THE App SHALL display a specific error message indicating the account is disabled
6. THE login form SHALL validate that the email field is not empty and has a valid email format before submitting
7. THE login form SHALL validate that the password field is not empty before submitting
8. WHILE login is in progress, THE login button SHALL display a loading indicator and be disabled

### Requirement 5: Email/Password Registration

**User Story:** As a dancer, I want to create an account with my email and password, so that I can start using the Dancee app.

#### Acceptance Criteria

1. WHEN the user submits the registration form with valid data, THE App SHALL call Auth_Cubit register with email, password, first name, and last name
2. WHEN registration succeeds, THE App SHALL automatically send an email verification link to the registered email address
3. WHEN registration succeeds, THE App SHALL navigate to the email verification screen (`/verify-email`)
4. IF registration fails because the email is already in use, THEN THE App SHALL display an error message indicating the email is already registered
5. IF registration fails due to a weak password, THEN THE App SHALL display an error message indicating the password requirements
6. THE registration form SHALL validate that first name and last name fields are not empty
7. THE registration form SHALL validate that the email field has a valid email format
8. THE registration form SHALL validate that the password is at least 8 characters long
9. THE registration form SHALL validate that the password and confirm password fields match
10. THE registration form SHALL require the user to accept the terms of use checkbox before submitting
11. WHILE registration is in progress, THE submit button SHALL display a loading indicator and be disabled

### Requirement 6: Google Sign-In

**User Story:** As a dancer, I want to sign in with my Google account, so that I can quickly access the app without creating a separate password.

#### Acceptance Criteria

1. WHEN the user taps the "Continue with Google" button on the login or register screen, THE App SHALL initiate the Google Sign-In flow
2. WHEN Google Sign-In completes successfully, THE App SHALL authenticate the Google credential with Firebase_Auth
3. WHEN Google Sign-In authentication succeeds and the user is new (first sign-in), THE App SHALL navigate to the onboarding screen (`/onboarding`)
4. WHEN Google Sign-In authentication succeeds and the user is returning, THE App SHALL navigate to the events page (`/events`)
5. IF the user cancels the Google Sign-In flow, THEN THE App SHALL remain on the current screen without displaying an error
6. IF Google Sign-In fails due to a network or server error, THEN THE App SHALL display an error message
7. WHILE Google Sign-In is in progress, THE Google button SHALL display a loading indicator and be disabled

### Requirement 7: Apple Sign-In

**User Story:** As a dancer using an Apple device, I want to sign in with my Apple ID, so that I can quickly access the app using my existing Apple account.

#### Acceptance Criteria

1. WHEN the user taps the "Continue with Apple" button on the login or register screen, THE App SHALL initiate the Apple Sign-In flow
2. WHEN Apple Sign-In completes successfully, THE App SHALL authenticate the Apple credential with Firebase_Auth
3. WHEN Apple Sign-In authentication succeeds and the user is new (first sign-in), THE App SHALL navigate to the onboarding screen (`/onboarding`)
4. WHEN Apple Sign-In authentication succeeds and the user is returning, THE App SHALL navigate to the events page (`/events`)
5. IF the user cancels the Apple Sign-In flow, THEN THE App SHALL remain on the current screen without displaying an error
6. IF Apple Sign-In fails due to a network or server error, THEN THE App SHALL display an error message
7. THE Apple Sign-In button SHALL only be visible on iOS and macOS platforms
8. WHILE Apple Sign-In is in progress, THE Apple button SHALL display a loading indicator and be disabled

### Requirement 8: Email Verification

**User Story:** As a dancer, I want to verify my email address after registration, so that my account is confirmed and secure.

#### Acceptance Criteria

1. WHEN the user arrives at the email verification screen, THE App SHALL display a message indicating that a verification email has been sent to the user's email address
2. THE email verification screen SHALL display the user's email address
3. THE email verification screen SHALL provide a "Resend verification email" button
4. WHEN the user taps "Resend verification email", THE App SHALL call Auth_Cubit sendEmailVerification and display a confirmation message
5. THE email verification screen SHALL provide a "I've verified my email" button that reloads the Firebase_User and checks the emailVerified status
6. WHEN the user taps "I've verified my email" and the email is verified, THE App SHALL navigate to the onboarding screen (`/onboarding`) for new users or the events page (`/events`) for returning users
7. WHEN the user taps "I've verified my email" and the email is not yet verified, THE App SHALL display a message indicating the email has not been verified yet
8. THE email verification screen SHALL provide a "Sign out" option so the user can return to the login screen


### Requirement 9: Password Reset

**User Story:** As a dancer, I want to reset my password if I forget it, so that I can regain access to my account.

#### Acceptance Criteria

1. WHEN the user submits the forgot password form with a valid email, THE App SHALL call Auth_Cubit sendPasswordReset with the email address
2. WHEN the password reset email is sent successfully, THE App SHALL display a success message and an info card explaining to check the inbox
3. IF the email address is not associated with any account, THE App SHALL still display the success message (to prevent email enumeration)
4. THE forgot password form SHALL validate that the email field is not empty and has a valid email format before submitting
5. WHILE the password reset request is in progress, THE submit button SHALL display a loading indicator and be disabled
6. THE forgot password screen SHALL provide a link to navigate back to the login screen

### Requirement 10: Protected Routing

**User Story:** As a developer, I want routes protected based on authentication state, so that unauthenticated users cannot access app content and authenticated users are redirected away from auth screens.

#### Acceptance Criteria

1. THE Router_Guard SHALL redirect unauthenticated users to the login screen (`/login`) when they attempt to access any protected route (events, courses, saved, profile, and their sub-routes)
2. THE Router_Guard SHALL redirect authenticated users with verified email away from auth screens (`/login`, `/register`, `/forgot-password`) to the events page (`/events`)
3. THE Router_Guard SHALL allow authenticated users with unverified email to access only the email verification screen (`/verify-email`) and redirect them there from other protected routes
4. THE Router_Guard SHALL allow all users to access the onboarding screen (`/onboarding`) only when authenticated
5. THE Router_Guard SHALL read the current Auth_State from the Auth_Cubit to determine redirect behavior
6. WHEN the Auth_State changes (e.g., user signs in or signs out), THE GoRouter SHALL re-evaluate the redirect and navigate accordingly

### Requirement 11: Sign Out

**User Story:** As a dancer, I want to sign out of the app, so that I can secure my account on shared devices.

#### Acceptance Criteria

1. WHEN the user taps "Log out" on the profile page, THE App SHALL call Auth_Cubit signOut
2. WHEN sign out completes, THE App SHALL navigate to the login screen (`/login`) and clear the navigation stack
3. THE Auth_Cubit SHALL sign out from Firebase_Auth and from Google Sign-In (if the user signed in via Google)
4. WHEN sign out completes, THE App SHALL clear any cached user-specific data (favorites state, loaded events/courses remain cached as they are public data)

### Requirement 12: Firebase-Directus User Synchronization

**User Story:** As a developer, I want Firebase users synchronized to Directus, so that CMS operations (favorites, future user data) use the real authenticated user identity.

#### Acceptance Criteria

1. WHEN a user successfully authenticates via Firebase_Auth (any method), THE App SHALL use the Firebase_User uid as the user identifier for all CMS operations (replacing the hardcoded Default_User_ID)
2. THE Favorites_Cubit SHALL read the current user's Firebase UID from the Auth_Cubit and use the UID when calling the Favorites_Repository methods
3. WHEN the user signs out, THE Favorites_Cubit SHALL clear the local favorites state
4. THE App SHALL pass the Firebase ID token as a Bearer token in the Authorization header of Directus API requests (replacing or supplementing the static access token) when a user is authenticated
5. IF the Firebase ID token expires, THEN THE App SHALL refresh the token automatically before making the next API request

### Requirement 13: Onboarding Flow After First Sign-In

**User Story:** As a new dancer, I want to go through an onboarding wizard after my first sign-in, so that the app can personalize my experience.

#### Acceptance Criteria

1. WHEN a user signs in for the first time (via any auth method), THE App SHALL navigate to the onboarding screen (`/onboarding`)
2. THE App SHALL determine first-time sign-in by checking the Firebase_User `metadata.creationTime` — if the creation time is within 60 seconds of the current time, the user is considered new
3. WHEN the user completes or skips the onboarding, THE App SHALL navigate to the events page (`/events`)
4. THE onboarding screen SHALL persist the user's selected dance style preferences and location to the CMS user profile (or local storage as a fallback)
5. WHEN a returning user signs in, THE App SHALL skip the onboarding and navigate directly to the events page (`/events`)

### Requirement 14: Auth Error Message Mapping

**User Story:** As a dancer, I want clear error messages when authentication fails, so that I understand what went wrong and how to fix it.

#### Acceptance Criteria

1. THE Auth_Repository SHALL map the Firebase_Auth error code `invalid-credential` to a message indicating incorrect email or password
2. THE Auth_Repository SHALL map the Firebase_Auth error code `user-disabled` to a message indicating the account has been disabled
3. THE Auth_Repository SHALL map the Firebase_Auth error code `email-already-in-use` to a message indicating the email is already registered
4. THE Auth_Repository SHALL map the Firebase_Auth error code `weak-password` to a message indicating the password is too weak
5. THE Auth_Repository SHALL map the Firebase_Auth error code `too-many-requests` to a message indicating too many attempts and to try again later
6. THE Auth_Repository SHALL map the Firebase_Auth error code `network-request-failed` to a message indicating a network connection problem
7. FOR ALL unmapped Firebase_Auth error codes, THE Auth_Repository SHALL return a generic error message indicating an unexpected error occurred
8. ALL error messages SHALL be defined as translation keys in the slang i18n files (en, cs, es) and accessed via the `t` global variable

### Requirement 15: Internationalization of Auth UI

**User Story:** As a user, I want all authentication screens fully translated, so that I can use the auth flow in my preferred language.

#### Acceptance Criteria

1. ALL new user-facing strings in auth screens (email verification screen, error messages, loading states, success messages) SHALL be defined in the slang i18n translation files (en, cs, es) and accessed via the `t` global variable
2. NO new user-facing string SHALL be hardcoded in Dart source code
3. WHEN new translation keys are added, THEY SHALL be added to all three language files: `strings.i18n.json` (en), `strings_cs.i18n.json` (cs), and `strings_es.i18n.json` (es)
4. THE existing auth translation keys (login, register, forgotPassword, onboarding, passwordStrength) SHALL be reused where applicable

### Requirement 16: Form Input Validation

**User Story:** As a dancer, I want real-time form validation on auth screens, so that I can correct mistakes before submitting.

#### Acceptance Criteria

1. THE login form SHALL display an inline error below the email field when the email format is invalid, after the field loses focus
2. THE login form SHALL display an inline error below the password field when the field is empty, after the field loses focus
3. THE registration form SHALL display an inline error below the first name and last name fields when they are empty, after the field loses focus
4. THE registration form SHALL display an inline error below the email field when the email format is invalid, after the field loses focus
5. THE registration form SHALL display an inline error below the password field when the password is shorter than 8 characters, after the field loses focus
6. THE registration form SHALL display an inline error below the confirm password field when the passwords do not match, after the field loses focus
7. THE registration form SHALL display a password strength indicator (weak, medium, strong, very strong) below the password field as the user types
8. THE forgot password form SHALL display an inline error below the email field when the email format is invalid, after the field loses focus
9. ALL validation error messages SHALL be defined as translation keys in the slang i18n files


### Requirement 17: Account Deletion

**User Story:** As a dancer, I want to delete my account from the app, so that I can remove my personal data and comply with my right to be forgotten (also required by Apple App Store guidelines for apps with account creation).

#### Acceptance Criteria

1. THE App SHALL provide a "Delete account" option on the profile screen, visually distinct as a destructive action
2. WHEN the user taps "Delete account", THE App SHALL display a confirmation dialog explaining that the action is permanent and all user data will be removed
3. WHEN the user confirms account deletion, THE App SHALL re-authenticate the user before proceeding (Firebase_Auth requires recent authentication for account deletion)
4. WHEN re-authentication succeeds, THE App SHALL delete the Directus_User record and associated user data (favorites, preferences) from the CMS before deleting the Firebase_Auth account
5. WHEN CMS data cleanup completes, THE App SHALL delete the Firebase_Auth account using the Firebase_User `delete()` method
6. WHEN account deletion completes successfully, THE App SHALL sign the user out, clear all cached user-specific data, and navigate to the login screen (`/login`)
7. IF account deletion fails at any step, THEN THE App SHALL display an error message explaining the failure and keep the user signed in
8. IF re-authentication fails (user enters wrong credentials or cancels), THEN THE App SHALL cancel the deletion process and display an appropriate message
9. THE Auth_Repository SHALL provide a method to re-authenticate the current user with their credential (email/password, Google, or Apple depending on the original sign-in provider)
10. THE Auth_Repository SHALL provide a method to delete the current Firebase_Auth account
11. ALL account deletion UI strings (button label, confirmation dialog text, success and error messages) SHALL be defined as translation keys in the slang i18n files (en, cs, es)
12. WHILE account deletion is in progress, THE App SHALL display a loading indicator and prevent the user from navigating away

### Requirement 18: Firebase Authentication Setup Documentation

**User Story:** As a developer, I want a comprehensive setup guide for Firebase Authentication, so that I can configure the entire auth stack from scratch after implementation is complete.

#### Acceptance Criteria

1. THE App SHALL include a setup guide document at `frontend/dancee_app2/docs/FIREBASE_AUTH_SETUP.md`
2. THE setup guide SHALL document Firebase project creation steps including enabling Authentication and the required sign-in providers (Email/Password, Google, Apple)
3. THE setup guide SHALL document FlutterFire CLI installation and usage for generating Firebase configuration files (`firebase_options.dart`)
4. THE setup guide SHALL document platform-specific configuration: placing `google-services.json` in the Android `app/` directory and `GoogleService-Info.plist` in the iOS `Runner/` directory
5. THE setup guide SHALL document Google Sign-In configuration including creating OAuth 2.0 client IDs in Google Cloud Console for Android (with SHA-1 fingerprint), iOS (with bundle ID), and web platforms
6. THE setup guide SHALL document Apple Sign-In configuration including enabling the Sign in with Apple capability in the Apple Developer portal, creating a Service ID, and configuring the return URL in Firebase Console
7. THE setup guide SHALL document Directus CMS configuration for Firebase auth integration including any required collections, fields, or access control rules for user synchronization
8. THE setup guide SHALL document all environment variables and `config.dart` entries required for Firebase Authentication, referencing `config.example.dart` as the template
9. THE setup guide SHALL be written in English following the project documentation standards
